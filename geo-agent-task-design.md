# Geo Agent Task Design

## Overview

基于 Meta ARE (Agent Research Environments) 框架，设计一套面向真实地理世界的 Agent Benchmark 评测任务。不同于对 ARE UI 智能体能力维度的简单映射，我们从**地理推理本身所需的认知能力**出发，通过第一性原理分析，提炼出 8 个面向地理空间推理的核心能力维度，并围绕这些能力设计评测任务。

**目标**: Benchmark 评测，量化 Agent 的地理推理能力
**数据接口**: Google Maps 全套 API (Directions, Places, Geocoding, Distance Matrix, Elevation, Traffic)
**模态**: 文本 + 图片（多模态输入）
**任务数量**: 23 个，分 5 个层级

---

## Capability Dimensions (能力维度)

### 设计理念

一个人在真实世界中完成地理任务时，到底在运用哪些认知能力？我们从这个问题出发，而非从"ARE 有哪些维度"出发，识别出以下 8 个地理推理的核心能力维度。

### 1. 空间认知 `SpatialCognition`

最基础、也最独特于地理推理的底层能力。

- **方位感知**: 理解东南西北、"向左/向右"、相对位置关系
- **距离估计**: 判断两点间的远近、"附近"到底多远
- **拓扑理解**: A 在 B 的里面/旁边/对面、路网的连通性
- **区域归属**: 中关村属于海淀区、浦东和浦西被黄浦江隔开

几乎所有地理任务都依赖这个能力，它是地理推理的根基。

### 2. 尺度判断 `ScaleReasoning`

人类对地理有强烈的尺度直觉，但 LLM 经常犯尺度错误。

- **可行性判断**: 30 分钟能从北京开车到上海吗？骑车 5km 要多久？
- **尺度切换**: 同一个"附近"，步行语境是 500m，驾车语境是 5km，航班语境是 500km
- **量级感知**: 海拔 5000m 意味着高原反应、距离 2000km 意味着不适合自驾一日游

这个能力直接决定了 Agent 会不会给出"荒谬"的答案。

### 3. 时空耦合推理 `SpatioTemporal`

地理推理中，空间和时间高度耦合，不可分割。

- **通勤时间估算**: 距离 + 交通方式 + 路况 → 到达时间
- **时间窗口约束**: 景点开放时间、会议截止时间、航班起飞时间
- **行程编排**: 在有限时间内访问多个地点，每个地点有自己的时间约束
- **时间流逝感知**: 现在是 11:30，吃饭要 1 小时，路上要 35 分钟，够不够？

这不是简单的"时间规划"，而是时间和空间的**联合推理**。

### 4. 路径规划与优化 `RoutePlanning`

地理中最经典的推理类型。

- **单路径规划**: A→B 的最优路线，考虑约束（避高速、避收费）
- **多点优化**: TSP 类问题，多个途经点的最优访问顺序
- **环形路线设计**: 出发 → 多点 → 返回，避免折返
- **多方案比较**: 驾车 vs 地铁 vs 骑行的权衡

### 5. 多源信息整合 `MultiSource`

真实地理任务往往需要组合多种异构信息源。

- **数据源串联**: 先地理编码（地址→坐标）→ 再搜索附近 → 再规划路线 → 再查海拔
- **偏好+地理交叉**: 组合用户偏好（川菜 + 评分>4.0）与地理约束（距离<3km）
- **跨模态转化**: 将视觉信息（图片中的地标）转化为可查询的地理信息

### 6. 动态环境适应 `DynamicAdaptation`

真实地理环境不是静态的。

- **突发事件**: 交通事故、道路施工、天气突变
- **需求变更**: 用户中途改变目的地、约束条件变化
- **状态记忆**: 记住之前停车的位置、之前规划的路线，在此基础上调整而非从头来过

关键在于 Agent 能否**增量式调整已有方案**。

### 7. 地理常识与纠错 `GeoCommonsense`

Agent 需要具备基本的地理常识来识别输入中的错误与歧义。

- **事实纠错**: 中关村不在朝阳区、海南没有地铁
- **噪声过滤**: 从冗长的闲聊中提取核心地理意图
- **歧义消解**: 全国有几十条中山路、"机场"是哪个机场

这是一种**鲁棒性**能力——不被错误输入和干扰信息带偏。

### 8. 视觉地理理解 `VisualSpatial`

地理场景天然涉及视觉信息，且不是通用的"看图说话"。

- **地标识别**: 从照片识别建筑/景点 → 定位到坐标
- **地图解读**: 读懂交通热力图、手绘路线图
- **街景推理**: 从街景照片判断方位和位置
- **地形理解**: 从地形图/卫星图理解地貌特征

核心是**从视觉到地理坐标/空间关系的映射**。

### 与 ARE UI 智能体能力的对比

```
ARE (UI Agent)              GeoAgent (地理推理)
──────────────              ──────────────────────
Execution        ←→        多源信息整合 MultiSource (更强调地理数据源串联)
Search           ←→        (融入空间认知和路径规划中，不单独成维度)
Ambiguity  ─┐
             ├──→          地理常识与纠错 GeoCommonsense (合并为地理特有的鲁棒性)
Noise      ─┘
Adaptability     ←→        动态环境适应 DynamicAdaptation (交通/天气/需求变更)
Time             ←→        时空耦合推理 SpatioTemporal (时间和空间深度绑定)
Agent2Agent      ←→        (地理场景中不太适用)
                 ──→        空间认知 SpatialCognition ★ 地理推理最核心的底层能力
                 ──→        尺度判断 ScaleReasoning ★ 防止荒谬答案
                 ──→        路径规划与优化 RoutePlanning ★ 经典地理推理
                 ──→        视觉地理理解 VisualSpatial ★ 跨模态空间映射
```

> 4 个维度（空间认知、尺度判断、路径优化、视觉地理）是**地理推理独有**的，ARE 中没有对应物。
> ARE 的 Ambiguity 和 Noise 在地理场景中合并为**地理常识与纠错**，因为歧义消解和噪声过滤在地理中都依赖同一种底层能力——对地理事实的掌握。

## Layer 1: Single Tool Call (基础能力)

### Task 1: 地址解析

- **Capabilities**: `SpatialCognition`
- **API**: Geocoding
- **Description**: 给定一个自然语言地址，返回经纬度坐标。

**示例输入**:

```
用户: 北京市海淀区中关村大街1号的经纬度是多少？
```

**示例输出**:

```
Agent: 北京市海淀区中关村大街1号的经纬度为：
- 纬度: 39.9836
- 经度: 116.3161
该位置位于中关村核心区域，靠近中关村地铁站。
```

**验证逻辑**: 返回的经纬度与标准答案误差 < 0.01

---

### Task 2: 两点距离查询

- **Capabilities**: `SpatialCognition`, `ScaleReasoning`
- **API**: Distance Matrix
- **Description**: 查询两点之间的距离和预估通勤时间。

**示例输入**:

```
用户: 从上海虹桥机场到浦东机场有多远？开车要多久？
```

**示例输出**:

```
Agent: 从上海虹桥机场到浦东机场的驾车信息：
- 距离: 约 56.3 公里
- 预计用时: 约 50 分钟（当前路况）
注意：实际用时会受交通状况影响，高峰期可能需要 1.5-2 小时。
```

**验证逻辑**: 返回了距离（km）和时长（min），且数值在合理范围内（距离 40-70km，时间 30-120min）

---

### Task 3: 地点搜索

- **Capabilities**: `SpatialCognition`
- **API**: Places (Nearby Search)
- **Description**: 在指定位置和半径内搜索特定类型的地点。

**示例输入**:

```
用户: 故宫附近1公里内有哪些咖啡厅？
```

**示例输出**:

```
Agent: 故宫（北京）附近1公里内找到以下咖啡厅：
1. 故宫角楼咖啡 - 评分 4.5 - 距离约 200m
2. Starbucks 东华门店 - 评分 4.2 - 距离约 650m
3. Manner Coffee 南池子店 - 评分 4.3 - 距离约 800m
...
```

**验证逻辑**: 返回结果全部在 1km 半径内，且类型为 cafe/coffee

---

### Task V1: 景点识别与定位

- **Capabilities**: `VisualSpatial`
- **API**: Geocoding
- **Description**: 根据一张景点图片，识别景点名称并返回地址和坐标。

**示例输入**:

```
用户: [附件: eiffel_tower.jpg] 这是哪里？给我它的地址和经纬度。
```

**示例输出**:

```
Agent: 这是法国巴黎的埃菲尔铁塔（Eiffel Tower）。
- 地址: Champ de Mars, 5 Av. Anatole France, 75007 Paris, France
- 纬度: 48.8584
- 经度: 2.2945
- 开放时间: 09:30-23:45（具体以官网为准）
```

**验证逻辑**: 正确识别景点名称，返回的经纬度与标准答案误差 < 0.01

---

### Task V2: 街景方向判断

- **Capabilities**: `VisualSpatial`, `SpatialCognition`
- **API**: Places, Directions
- **Description**: 根据一张街景照片，推断用户所在位置和朝向，并给出到最近目标的方向引导。

**示例输入**:

```
用户: [附件: street_view.jpg —— 照片中可见"王府井大街"路牌和东方新天地商场]
     我现在站在这里，最近的地铁站怎么走？
```

**示例输出**:

```
Agent: 根据照片中的"王府井大街"路牌和东方新天地商场，您应该在王府井大街南端。
最近的地铁站是 王府井站（1号线），步行约 3 分钟：
1. 沿王府井大街向北步行 200 米
2. 在路东侧可看到地铁入口（A口）
```

**验证逻辑**: 正确识别位置（王府井区域），推荐的地铁站合理，步行路线方向正确

## Layer 2: Multi-Step Reasoning (组合能力)

### Task 4: 基础路线规划

- **Capabilities**: `RoutePlanning`
- **API**: Directions
- **Description**: 规划满足特定约束的路线（如避开高速、避开收费站）。

**示例输入**:

```
用户: 帮我规划从北京大学到天安门的驾车路线，避开高速。
```

**示例输出**:

```
Agent: 从北京大学到天安门的驾车路线（已避开高速）：
- 总距离: 14.2 公里
- 预计用时: 约 35 分钟

路线：
1. 从北京大学东门出发，沿中关村北大街向南
2. 右转进入海淀南路
3. 沿西二环辅路向南
4. 经西长安街到达天安门

注意：该路线全程走城市道路，不经过高速公路。
```

**验证逻辑**: 返回的路线中不包含高速公路路段，起终点正确

---

### Task 5: 多模式出行对比

- **Capabilities**: `RoutePlanning`, `ScaleReasoning`
- **API**: Directions (driving/transit/bicycling)
- **Description**: 比较多种出行方式的时间、距离、费用，给出合理推荐。

**示例输入**:

```
用户: 从西二旗地铁站到国贸，开车、地铁、骑车分别要多久？推荐哪种？
```

**示例输出**:

```
Agent: 从西二旗到国贸的三种出行方式对比：

| 方式 | 距离 | 用时 | 费用估算 |
|------|------|------|----------|
| 驾车 | 22km | 45-80min | 停车费约30-50元 |
| 地铁 | - | 约55min | 7元 |
| 骑车 | 24km | 约90min | 免费 |

推荐：地铁。
理由：虽然驾车最快（不堵车时），但早晚高峰西二旗到国贸方向严重拥堵，
实际用时可能超过80分钟。地铁用时稳定，且无需考虑停车问题。
```

**验证逻辑**: 三种方式都查询了，推荐结果包含合理依据（对比分析）

---

### Task 6: 途经点路线优化

- **Capabilities**: `RoutePlanning`
- **API**: Directions (waypoints + optimize)
- **Description**: 给定多个途经点，规划最优访问顺序，避免折返。

**示例输入**:

```
用户: 我要从酒店（王府井）出发，先去故宫，再去颐和园，最后去首都机场，帮我规划最优驾车路线。
```

**示例输出**:

```
Agent: 最优驾车路线（已优化途经顺序）：

王府井酒店 → 故宫（5min, 2km）
  → 颐和园（40min, 18km）
  → 首都机场 T3（45min, 32km）

总距离: 约 52 公里
总用时: 约 1 小时 30 分钟

优化说明：按地理位置从市中心向西北再向东北，避免来回折返。
原始顺序和优化顺序一致，无需调整。

提示：故宫周一闭馆，请注意安排日期。
```

**验证逻辑**: 所有途经点都包含在路线中，总路程优于未优化的顺序

---

### Task 13: 同名地点消歧

- **Capabilities**: `GeoCommonsense`
- **API**: Geocoding, Places
- **Description**: 用户提及的地点存在多个同名候选，Agent 需识别歧义并主动澄清或基于上下文推断。
- **评估能力**: 测试 Agent 是否具备"全国有几十条中山路"这类地理常识，并在信息不足时主动向用户请求澄清。

**示例输入**:

```
用户: 帮我导航到中山路上的那家星巴克。
```

**示例输出**:

```
Agent: "中山路"在多个城市都有，且大多数城市的中山路上有不止一家星巴克。
请问您指的是哪个城市？

或者，如果您能告诉我您当前所在的城市，我可以帮您搜索最近的中山路星巴克。

常见的中山路：
1. 厦门市中山路（步行街，有 2 家星巴克）
2. 南京市中山路（有 3 家星巴克）
3. 广州市中山路（有多家星巴克）
```

**验证逻辑**: Agent 识别到"中山路"存在歧义，未直接给出单一结果，而是主动向用户请求澄清信息

---

### Task 14: 错误地址纠正

- **Capabilities**: `GeoCommonsense`
- **API**: Geocoding, Places
- **Description**: 用户提供的地址包含错误信息（区域张冠李戴），Agent 需凭地理常识识别并纠正。
- **评估能力**: 测试 Agent 是否知道"中关村在海淀区不在朝阳区"这类地理事实，能否在输入有误时仍返回正确结果。

**示例输入**:

```
用户: 帮我查一下"北京市朝阳区中关村大街27号"的位置信息。
```

**示例输出**:

```
Agent: 注意：中关村大街位于北京市海淀区，而非朝阳区。
您可能指的是以下地址之一：

1. 北京市海淀区中关村大街27号（中关村核心区域）
   - 纬度: 39.9817 | 经度: 116.3118
   - 附近地标: 中关村购物广场

2. 如果您确实要查朝阳区的地址，请提供更多信息。

已按"海淀区中关村大街27号"为您查询。
```

**验证逻辑**: Agent 正确识别了区域错误（朝阳→海淀），返回了纠正后的正确结果，而非直接报错或返回错误位置

### Task V3: 相似景点推荐

- **Capabilities**: `VisualSpatial`, `SpatialCognition`
- **API**: Places
- **Description**: 根据图片识别景点特征，在用户指定区域推荐相似地点。

**示例输入**:

```
用户: [附件: karst_landscape.jpg —— 桂林漓江山水照片]
     我不知道这是哪个景点，但我很喜欢这种风格，我现在在成都，想在周边找个类似的地方旅行。
```

**示例输出**:

```
Agent: 这张图片是广西桂林漓江的喀斯特地貌山水风景。

这种地貌的特征是：石灰岩溶蚀形成的峰林、溶洞和碧绿水域。
在成都周边，推荐以下类似风景的目的地：

1. 蜀南竹海（宜宾）- 距成都约 280km，车程 3.5h
   - 特色：竹林覆盖的喀斯特丘陵，有溶洞和溪流

2. 兴文石海（宜宾）- 距成都约 350km，车程 4h
   - 特色：典型喀斯特地貌，石林、溶洞、天坑

3. 九寨沟（阿坝）- 距成都约 440km，车程 7h
   - 特色：虽非喀斯特，但有类似的碧绿水域和山水意境

推荐首选蜀南竹海，距离较近且最接近图片中的山水风格。
```

**验证逻辑**: 正确识别图片中的地貌特征（喀斯特/山水），推荐地点确实在成都周边（< 500km），且具有相似的自然特征

## Layer 3: Constraint Reasoning (条件约束)

### Task 7: 时间约束规划

- **Capabilities**: `SpatioTemporal`, `MultiSource`
- **API**: Places, Distance Matrix, Directions
- **Description**: 在时间约束下规划"吃饭 + 赴约"的行程，需要联合推理时间窗口与空间距离。

**示例输入**:

```
用户: 我下午2点在三里屯有个会议，现在（11:30）在清华大学。
     我想先在附近吃个午饭再过去，推荐一个餐厅并规划路线，确保我能准时到。
```

**示例输出**:

```

- 可用时间: 11:30 - 14:00 = 2.5 小时
```
